# Система обработки ошибок API

## Обзор

В проекте реализована централизованная система обработки ошибок API, которая автоматически отображает понятные сообщения об ошибках на русском языке через toast-уведомления.

## Компоненты

### 1. `errorHandler.js` - Утилита обработки ошибок

Основной файл, содержащий:
- **HTTP_ERROR_MESSAGES** - Сообщения для HTTP статусов
- **ENDPOINT_ERROR_MESSAGES** - Специфические сообщения для эндпоинтов
- **handleApiError()** - Главная функция обработки ошибок
- **handleSuccess(), handleWarning(), handleInfo()** - Вспомогательные функции

### 2. `toast.js` - Система уведомлений

Отвечает за отображение всплывающих уведомлений:
- Автоматическое создание контейнера
- Анимированное появление/исчезновение
- Различные типы (success, error, warning, info)
- Автоматическое удаление через 3 секунды

### 3. `client.js` - API клиент

Улучшенный API клиент с интегрированной обработкой ошибок:
- Автоматический вызов `handleApiError()` при ошибках
- Поддержка FormData для загрузки файлов
- Обновление токенов авторизации

## Как это работает

### Автоматическая обработка ошибок

При любом API запросе:
1. Если запрос неудачный, API клиент автоматически вызывает `handleApiError()`
2. Функция определяет тип ошибки по статус-коду и эндпоинту
3. Показывается toast-уведомление с русским сообщением
4. Ошибка все еще выбрасывается для дополнительной обработки в компонентах

### Пример использования

```javascript
// Раньше нужно было вручную обрабатывать ошибки:
try {
  await projects.create(projectData)
} catch (err) {
  setError(err.message) // Ручная обработка
}

// Теперь ошибки обрабатываются автоматически:
try {
  await projects.create(projectData) // Ошибка автоматически покажется в toast
} catch (err) {
  // Только для специфической логики компонента
  if (!projectData.title) {
    setError('Название обязательно')
  }
}
```

## Сообщения об ошибках

### Общие HTTP ошибки
- **400** - "Ошибка валидации. Проверьте правильность введенных данных"
- **401** - "Требуется авторизация. Войдите в систему"
- **403** - "Нет доступа к этому ресурсу"
- **404** - "Запрошенный ресурс не найден"
- **500** - "Внутренняя ошибка сервера"

### Специфические ошибки эндпоинтов

#### Аутентификация
- `/users/login/` (401) - "Неверный email или пароль"
- `/users/verify-code/` (400) - "Код истёк, неверный код или email уже подтверждён"

#### Проекты
- `/projects/{id}/` (404) - "Проект не найден"
- `/projects/{project_id}/upload/` (400) - "Ошибка загрузки файлов. Проверьте формат и размер"

#### Чаты
- `/chats/{chat_id}/ask/` (400) - "Отсутствует текст сообщения"
- `/chats/{chat_id}/ask/` (403) - "Нет доступа к чату"

## Ручное использование

Если нужно вручную показать уведомление:

```javascript
import { handleSuccess, handleWarning, handleInfo } from '../utils/errorHandler.js'

// Успешное действие
handleSuccess('Проект успешно создан')

// Предупреждение
handleWarning('Файл слишком большой')

// Информационное сообщение
handleInfo('Загрузка началась')
```

## Преимущества

1. **Централизация** - Все ошибки обрабатываются в одном месте
2. **Консистентность** - Единый стиль сообщений
3. **Локализация** - Все сообщения на русском языке
4. **Автоматизация** - Минимум кода в компонентах
5. **Гибкость** - Возможность переопределения при необходимости
6. **UX** - Ненавязчивые toast-уведомления

## Добавление новых ошибок

Для добавления нового сообщения об ошибке:

```javascript
// В ENDPOINT_ERROR_MESSAGES
'/new/endpoint/': {
  400: 'Специфическое сообщение для этого эндпоинта',
  404: 'Ресурс не найден'
}
```

## Тестирование

Для тестирования обработки ошибок можно использовать Network tab в DevTools и установить разные HTTP статусы или симулировать ошибки сервера.
